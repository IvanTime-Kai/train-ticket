── api flow ──

// end-to-end booking flow
1. Hold Seat (giữ ghế)
  • 01POST /bookings/hold {trip_id, seat_ids[]}
  • 02gRPC → train-service: ValidateSeat(trip_id, seat_id)
  • 03Ghế tồn tại & thuộc trip → OK
  • 04Redis SET NX seat_hold:tripID:seatID → TTL 5 phút
  • 05Đã có người hold → reject "seat already held"
  • 06Trả về hold_token (TTL 5 phút)
2. Create Booking
  • 01POST /bookings {trip_id, seat_ids[], hold_token}
  • 02Verify hold_token trong Redis còn không
  • 03Hết hạn → reject "hold expired"
  • 04gRPC → train-service: GetSeatPrice(seat_ids[])
  • 05Lưu booking (PENDING) + booking_seats vào DB
  • 06Kafka publish: booking.created
  • 07Trả về booking info + payment URL
3. Payment Success
  • 01Kafka consume: payment.success
  • 02Update booking status → CONFIRMED
  • 03Xoá seat_hold khỏi Redis
  • 04Kafka publish: booking.confirmed
  • 05notification-service gửi email xác nhận
4. Cancel / Expire Booking
  • 01DELETE /bookings/:id (user cancel)
  • 02Update booking status → CANCELLED
  • 03Xoá seat_hold khỏi Redis → ghế tự do
  • 04Kafka publish: booking.cancelled
  • ──── Auto expire ──
  • 05Redis TTL hết → seat_hold tự xoá
  • 06Cron job → booking PENDING quá 15 phút → EXPIRED

── redis key design ──

// Redis Keys
seat_hold:<tripID>:<seatID>
value: userID
TTL: 5 phút
dùng: SET NX (chỉ set nếu chưa có)
→ Tránh 2 người hold cùng ghế
hold_token:<token>
value: {userID, tripID, seatIDs[]}
TTL: 5 phút
dùng: verify khi tạo booking
→ Đảm bảo user đã hold ghế trước

── kafka topics ──

// Event-Driven Communication

payment.success
publisher: payment-service

booking.confirmed
publisher: booking-service
